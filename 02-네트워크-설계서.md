## 1. 전체 요청 흐름

### 결정사항

모든 트래픽은 인터넷 → API Gateway → Lambda → DynamoDB 단방향 경로로 흐른다. 

```
[Flutter 앱]
    │  HTTPS (TLS 1.2+)
    ▼
[API Gateway HTTP API v2]  ─── 공개 라우트 (5개): Authorizer 없음
    │                      ─── 보호 라우트 (12개): Lambda Authorizer 선행 실행
    │
    ├─ 공개 요청 ──────────────────────────────────────────────────────────┐
    │                                                                      │
    ├─ 보호 요청 ──► [Lambda: authorizer]                                  │
    │                    │ JWT 검증 성공 → userId 컨텍스트 주입             │
    │                    │ JWT 검증 실패 → 403 반환 (Lambda 미호출)         │
    │                    ▼                                                  │
    └────────────► [Lambda: 비즈니스 함수] ◄──────────────────────────────┘
                        │  boto3 (AWS SDK)
                        │  HTTPS (AWS 내부 TLS)
                        ▼
                  [DynamoDB]
                  (Users / SeatRegistrations / TransactionLogs)
```

**인증이 필요 없는 공개 라우트 5개**

| 라우트 | 이유 |
|--------|------|
| `POST /api/v1/auth/signup` | 로그인 전 단계이므로 토큰 없음 |
| `GET /api/v1/seats` | 열차 좌석 목록 조회는 비로그인 사용자도 열람 가능 |
| `GET /api/v1/seats/all` | 마케팅용 전체 좌석 현황은 비인증 사용자에게도 공개 |
| `GET /api/v1/train/identify` | 실시간 열차 위치 조회는 공개 정보 |
| `GET /api/v1/arrival/nearby` | 역 도착 정보는 역 이름만 있으면 누구나 조회 가능한 공공 데이터 |

**`get_train` , `get_nearby_arrival` 함수의 외부 API 호출**

서울시 실시간 지하철 API를 추가로 호출한다. 공개되지 않은 라우트에 해당하는 `pass_seat` 함수도 같은 외부 API를 호출한다.

```
[Lambda]
    │  HTTP 아웃바운드
    ▼
[서울시 실시간 지하철 API]
(외부 공공 API, SEOUL_API_KEY 인증)
```

**`auth` 함수의 외부 API 호출**

Kakao 및 Apple 인증 처리를 위해 외부 API를 호출한다. Apple 로그인의 경우 JWKS 엔드포인트에서 공개키를 가져오는 과정에서 지연이 발생할 수 있어 타임아웃을 15초로 설정했다.

```
[Lambda: auth]
    ├─ Kakao: GET https://kapi.kakao.com/v1/user/access_token_info
    └─ Apple: GET https://appleid.apple.com/auth/keys (JWKS)
```

## 2. API Gateway 구성

### 결정사항

AWS API Gateway **HTTP API (v2)** 를 사용한다. REST API (v1)를 사용하지 않는다.

```hcl
resource "aws_apigatewayv2_api" "main" {
  name          = "${local.prefix}-api"
  protocol_type = "HTTP"
}
```

**API Gateway 설정**

| 항목 | 값 | 비고 |
|------|-----|------|
| 프로토콜 | HTTP | WebSocket 미사용 |
| 엔드포인트 타입 | Regional (기본값) | 서울 리전 단일 운영 |
| 스테이지 | `$default` | 단일 스테이지, `auto_deploy = true` |
| 통합 방식 | `AWS_PROXY` | Lambda Proxy 통합 |
| Payload Format | `2.0` | HTTP API v2 전용 포맷 |
| Authorizer 타입 | `REQUEST` (Lambda) | JWT 커스텀 검증 |
| Authorizer Payload | `2.0` | `enable_simple_responses = false` |
| 액세스 로그 | CloudWatch Logs | 14일 보존 |

**스테이지 구성**

```hcl
resource "aws_apigatewayv2_stage" "default" {
  name        = "$default"
  auto_deploy = true
  # 액세스 로그: requestId, 메서드, 경로, 상태코드, 응답크기, 지연시간, UA, IP, 에러메시지
}
```

`auto_deploy = true`로 설정하여 라우트/통합 변경 시 수동 배포 없이 즉시 반영된다.

**Lambda Authorizer 동작 방식**

```
요청 수신
    │
    ▼
[Authorizer Lambda 호출]
    │
    ├─ 성공: IAM Policy 반환 + userId를 컨텍스트에 주입
    │         → 비즈니스 Lambda 호출
    │         → event["requestContext"]["authorizer"]["lambda"]["userId"] 로 접근
    │
    └─ 실패: 403 Forbidden 반환
              → 비즈니스 Lambda 미호출
```

### API Gateway 타입 선택 근거 (HTTP API vs REST API)

REST API가 제공하는 기능 중 현재 서비스에 필요하지 않고 사용 계획도 없는 항목들이 있어 HTTP API를 선택했다.
추가적으로 HTTP API는 REST API보다 응답 지연이 낮다. AWS 공식 문서는 최대 60% 개선을 주장하나, 실측 기준으로는 약 14~16%(절대값 약 2.5ms) 수준이다.

| REST API 전용 기능 | 현재 필요 여부 | 판단 근거 |
|-------------------|--------------|----------|
| API Key 기반 접근 제어 | ❌ | 모바일 앱 전용 서비스로 서드파티 API 제공 계획 없음 |
| 게이트웨이 레벨 응답 캐싱 | ❌ | 실시간 지하철 위치 데이터는 캐싱이 오히려 부적합 (단 Authorizer **결과** 캐싱은 별개 기능으로 적용함 → 05-보안-설계서 참조) |
| WAF 통합 | ❌ | 필요성은 인정하나, 현 시점에서는 필요가 없다고 판단 |
| Private 엔드포인트 (VPC 내부 전용) | ❌ | VPC 미사용 아키텍처 |
| 요청/응답 변환 | ❌ | Lambda Proxy 통합으로 변환 로직을 Lambda에서 직접 처리 |
| 사용량 계획 | ❌ | 외부 개발자 대상 API 제공 계획 없음 |

### 트레이드오프

HTTP API를 선택함으로써 REST API가 기본으로 제공하는 일부 보안 기능을 직접 구성해야 한다.

**WAF**: HTTP API는 WAF 직접 통합을 지원하지 않는다. 공격 대응이 필요해질 경우 CloudFront를 앞단에 추가하고 WAF를 직접 연결하는 별도 구성이 필요하다.

## 3. Lambda ↔ DynamoDB 통신 경로

### 결정사항

Lambda는 `boto3` AWS SDK를 통해 DynamoDB 퍼블릭 엔드포인트에 HTTPS로 직접 통신한다. 

```
[Lambda 함수]
    │
    │  boto3.resource("dynamodb", region_name="ap-northeast-2")
    │  HTTPS (TLS) — AWS 퍼블릭 엔드포인트
    │  dynamodb.ap-northeast-2.amazonaws.com
    ▼
[DynamoDB]
```

**테이블 참조 방식**

Lambda는 환경변수로 테이블 이름을 주입받아 사용한다. 테이블 이름을 코드에 하드코딩하지 않는다.

```python
# utils.py — 공유 레이어
users_table              = _dynamodb.Table(os.getenv("USERS_TABLE"))
seat_registrations_table = _dynamodb.Table(os.getenv("SEAT_REGISTRATIONS_TABLE"))
transaction_logs_table   = _dynamodb.Table(os.getenv("TRANSACTION_LOGS_TABLE"))
```

```hcl
# lambda.tf — Terraform이 테이블 이름을 환경변수로 주입
common_env = {
  USERS_TABLE              = aws_dynamodb_table.users.name
  SEAT_REGISTRATIONS_TABLE = aws_dynamodb_table.seat_registrations.name
  TRANSACTION_LOGS_TABLE   = aws_dynamodb_table.transaction_logs.name
}
```

**참고**

- boto3는 기본적으로 HTTPS(TLS)를 사용하여 DynamoDB와 통신한다.
- 인증은 Lambda 실행 역할의 IAM 자격증명으로 처리된다. 별도의 액세스 키를 코드에 포함하지 않는다.
- IAM Policy에서 허용된 테이블 ARN 외의 DynamoDB 리소스에는 접근할 수 없다.
