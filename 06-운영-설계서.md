## 1. CloudWatch 로그 구성

### 결정사항

API Gateway 액세스 로그와 Lambda 실행 로그를 모두 CloudWatch Logs에 수집한다. 
모든 로그 그룹의 보존 기간은 14일로 통일한다.

**API Gateway 액세스 로그**

```hcl
resource "aws_cloudwatch_log_group" "api_gateway" {
  name              = "/aws/apigateway/{service}-api"
  retention_in_days = 14
}
```

수집 필드:

| 필드 | 설명 |
|------|------|
| `requestId` | 요청 고유 ID (디버깅 추적용) |
| `requestTime` | 요청 수신 시각 |
| `httpMethod` | HTTP 메서드 |
| `routeKey` | 매칭된 라우트 (예: `POST /api/v1/seats`) |
| `path` | 실제 요청 경로 |
| `status` | HTTP 응답 상태 코드 |
| `responseLength` | 응답 바이트 크기 |
| `latency` | API Gateway 처리 지연시간 (ms) |
| `userAgent` | 클라이언트 User-Agent |
| `sourceIp` | 클라이언트 IP |
| `errorMessage` | API Gateway 레벨 에러 메시지 |

**Lambda 실행 로그**

모든 Lambda 함수에 대해 CloudWatch Log Group을 Terraform으로 명시적으로 생성한다. 로그 그룹 이름은 `/aws/lambda/{service}-{function-name}` 규칙을 따른다.

```hcl
resource "aws_cloudwatch_log_group" "register_seat" {
  name              = "/aws/lambda/{service}-register-seat"
  retention_in_days = 14
}
```

**Terraform으로 로그 그룹을 명시적으로 생성하는 이유**

Lambda는 첫 실행 시 로그 그룹을 자동으로 생성하지만, 이 경우 보존 기간이 무제한(Never expire)으로 설정된다. Terraform으로 미리 생성하면 보존 기간을 14일로 제어할 수 있고, 로그 그룹이 인프라 코드로 관리된다.

**보존 기간 14일 선택 근거**

실시간으로 제공해야 하는 지하철 좌석 공유 서비스의 특성상 장기 로그 보존의 필요성이 낮다고 판단했다. 
또한 무제한 보존 시 CloudWatch Logs 스토리지 비용이 누적된다. (GB당 $0.03/월).

## 2. 모니터링 항목

### 결정사항

별도의 알람(SNS, 이메일 등)은 미구성이다. CloudWatch 콘솔에서 수동으로 확인한다.

**현재 확인 가능한 지표 (CloudWatch 기본 제공)**

Lambda 함수별로 아래 지표가 자동 수집된다:

| 지표 | 설명 |
|------|------|
| `Invocations` | 함수 호출 횟수 |
| `Errors` | 에러 발생 횟수 |
| `Duration` | 실행 시간 (ms) |
| `Throttles` | 동시 실행 한도 초과 횟수 |
| `ConcurrentExecutions` | 동시 실행 수 |

API Gateway 기본 지표:

| 지표 | 설명 |
|------|------|
| `Count` | 총 요청 수 |
| `4XXError` | 클라이언트 에러 수 |
| `5XXError` | 서버 에러 수 |
| `Latency` | 전체 응답 지연시간 |
| `IntegrationLatency` | Lambda 실행 지연시간 |

**로그 기반 디버깅 방법**

에러가 발생한 기능을 파악한 뒤 CloudWatch 콘솔에서 해당 Lambda 함수의 로그 그룹(`/aws/lambda/{service}-{function-name}`)에 직접 접근하여 시간대 기준으로 로그를 확인한다.

## 3. 알람

### 결정사항

미구성. 향후 구성 예정.

### 구성 예정 알람

| 알람 | 지표 | 대상 | 이유 |
|------|------|------|------|
| 로그인 에러 | `Errors` | `auth` | 터지면 아무도 앱 접속 불가 |
| 인증 에러 | `Errors` | `authorizer` | 터지면 보호 라우트 전체 차단 |
| 외부 API 지연 | `Duration` > 2,500ms | `get_train`, `get_nearby_arrival`, `pass_seat` | 서울시 API 지연/장애 조기 감지 |
| 타임아웃 위험 (일반) | `Duration` > 2,500ms | 전체 (timeout 3s 함수) | timeout 직전 감지 |
| 타임아웃 위험 (auth) | `Duration` > 12,000ms | `auth`, `delete_user` | Apple JWKS 지연으로 인한 timeout 감지 |
| 사용자 노출 에러 | `5XXError` | API Gateway | 실제로 사용자가 에러를 받고 있다는 신호 |

## 4. 백업 전략

### 결정사항

미구성. 향후 구성 예정.

Users, TransactionLogs 테이블은 사용자 정보와 포인트 내역을 담고 있어 실사용자가 늘어나면 PITR 활성화를 권장한다. 설정은 한 줄이다.

```hcl
point_in_time_recovery {
  enabled = true
}
```
