# 거버넌스 아키텍처 설계서

> 최종 수정일: 2026-06-02



## 1. 리소스 네이밍 규칙

### 결정사항

모든 AWS 리소스 이름에 `{service}` 프리픽스를 적용한다.

```hcl
# infra/locals.tf
locals {
  prefix = "{service}"
}
```

적용 예시:

| 리소스 유형 | 실제 이름 |
|------------|----------|
| IAM Role | `{service}-lambda-role` |
| IAM Policy | `{service}-lambda-dynamodb-policy` |
| DynamoDB 테이블 | `{service}-Users`, `{service}-SeatRegistrations`, `{service}-TransactionLogs` |
| API Gateway | `{service}-api` |
| Lambda 함수 | `{service}-register-seat`, `{service}-get-train` 등 |

Lambda 함수명은 Terraform 리소스명(언더스코어)과 AWS 실제 이름(하이픈)이 다르다.  
예: Terraform `register_seat` → AWS `{service}-register-seat`



## 2. IAM Role/Policy 설계

### 결정사항

모든 Lambda 함수가 단일 실행 역할(`{service}-lambda-role`)을 공유한다.

**역할 구성**

```
{service}-lambda-role
├── AWSLambdaBasicExecutionRole  (AWS 관리형 정책 — CloudWatch Logs 쓰기)
└── {service}-lambda-dynamodb-policy  (커스텀 정책)
```

**DynamoDB 커스텀 정책 허용 액션**

```json
"Action": [
  "dynamodb:GetItem",
  "dynamodb:PutItem",
  "dynamodb:UpdateItem",
  "dynamodb:DeleteItem",
  "dynamodb:Query",
  "dynamodb:Scan"
]
```

허용 리소스는 세 테이블 ARN과 `SeatRegistrations` GSI ARN으로 명시적으로 제한한다.  
`Resource: "*"` 와일드카드를 사용하지 않는다.

### 최소 권한 원칙(Least Privilege) 적용 여부

**부분 적용** - 리소스 범위는 최소화했으나, 액션 범위는 완전한 최소 권한이 아니다.

- ✅ **리소스 범위**: 특정 테이블 ARN만 허용. 계정 내 다른 DynamoDB 테이블에 접근 불가.
- ✅ **서비스 범위**: DynamoDB와 CloudWatch Logs만 허용. S3, SQS 등 미사용 서비스 접근 없음.
- ⚠️ **액션 범위**: `dynamodb:Scan`이 포함되어 있으나 실제 코드에서는 Scan을 사용하지 않는다 (GSI Query로 대체). 불필요한 권한이 존재한다.
- ⚠️ **함수별 역할 미분리**: 예를 들어 `get_train` 함수는 DynamoDB 쓰기 권한이 불필요하지만 공유 역할로 인해 보유한다.

### 결정 근거

초기에는 넓은 권한을 부여해두고, 실제 사용 패턴이 파악되면 함수별로 필요한 권한만 남기는 방향으로 점진적으로 줄여나갈 예정이다. 리소스 ARN은 명시적으로 제한하여 최소한의 리소스 범위는 확보했다.

### 트레이드오프

| 현재 방식 (단일 공유 역할) | 개선 방향 (함수별 역할) |
|--------------------------|----------------------|
| Terraform 코드 단순 | 보안 강화 (읽기 전용 함수에 쓰기 권한 없음) |
| 역할 관리 포인트 1개 | 함수 수만큼 역할 관리 필요 |
| 침해 시 모든 테이블 접근 가능 | 침해 범위를 함수 단위로 격리 |



## 3. Tagging 전략

### 결정사항

표준화된 태깅 전략은 미적용 상태다. 단일 프로젝트로 비용 배분, 팀 소유권 추적, 환경 구분의 필요성이 없어 태깅을 최소화했다.



## 4. 인프라 관리 방식

### 결정사항

AWS 인프라 전체를 Terraform으로 코드화한다. AWS 콘솔 수동 조작을 사용하지 않는다.



## 5. 단일 환경 결정 근거와 트레이드오프

### 결정사항

dev/staging/prod 환경을 분리하지 않는다. 단일 환경(`production`)만 운영한다.

현재 구조:
- API Gateway: 단일 `$default` 스테이지 (`auto_deploy = true`)
- DynamoDB: 환경 접두사 없는 단일 테이블 세트
- Lambda: 단일 함수 세트

### 결정 근거

1. **프로젝트 규모**: 초기 런칭 단계로 트래픽 규모가 예측 불가능하며,  환경 분리 없이도 서버리스 아키텍처 특성상 자동 확장이 가능하다.
2. **관리 복잡도**: Terraform workspace 또는 별도 디렉토리 구조 도입 시 관리 포인트가 증가한다.
3. **배포 전략**: CI/CD에서 변경된 Lambda 함수만 선택적으로 배포하는 방식으로 위험을 최소화한다 (`git diff`로 변경 함수 감지).
4. **앱 배포 타이밍과 백엔드 배포를 맞추기 어려움**: 서비스 초기에는 API 응답 구조나 필드명이 자주 바뀐다. 예를 들어 `carNo`라는 필드를 `car_number`로 변경하면, 새 백엔드를 prod에 먼저 올리는 순간 아직 구버전 앱을 쓰는 사용자들은 `carNo`를 찾지 못해 에러가 발생한다. 반대로 새 필드를 추가하는 정도의 변경은 구버전 앱이 무시하면 그만이라 사전 배포가 가능하지만, 필드명 변경이나 응답 구조 개편처럼 기존 앱과 맞지 않는 변경은 새 앱이 배포되는 시점에 맞춰 백엔드도 함께 올려야 한다. 이 경우 iOS App Store와 Google Play의 심사 완료 시점이 달라 두 플랫폼 모두 승인될 때까지 기다렸다가 백엔드 배포와 앱 게시를 동시에 조율해야 하는 운영 부담이 생긴다.
5. **팀 의사결정**: 빠른 출시가 최우선 목표였다. 팀원들과 논의한 결과 현 단계에서 환경 분리는 관리 복잡도 대비 실질적인 이득이 없다고 판단했다. 서비스 안정화 이후 환경 분리를 고려한다.

### 개선해야 할 점

- **안전성**: 코드 버그가 즉시 실사용자에게 영향을 준다. 
  - 개선안: 
    Lambda 별칭(alias)과 가중치 기반 트래픽 분산을 활용한 Canary 배포?
- **롤백**: 문제 발생 시 이전 코드로 수동 롤백이 필요하다. 
  - 개선안:
    Lambda 버전 관리와 별칭을 활용하여 이전 버전으로 즉시 전환?
