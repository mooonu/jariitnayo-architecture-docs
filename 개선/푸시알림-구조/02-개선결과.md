# 푸시 알림 구조 개선 결과 - `notification_worker` 분리

[2026.06.17]

<br>

## 0. 요약

- **푸시 발송 자체 성공**: cold start에서 15초 타임아웃으로 실패하던 푸시 발송이 4초 이내 정상 완료 (128MB 동기 호출 → 512MB worker 분리)
- 기존 비즈니스 람다(`confirm_seat` / `pass_seat`)는 응답시간·메모리 영향 없이 운영 (128MB 유지, 응답 ~50ms 수준)
- 무거운 의존성(`firebase_admin`)과 Firebase 자격증명을 worker 한 곳에 격리

<br>

## 1. 적용 사항

<br>

### 람다

- `lambda/functions/test_push/` → `lambda/functions/notification_worker/` 디렉토리 이름 변경
- 핸들러 리팩토링: JWT 인증 제거, 요청 페이로드(`userId`, `title`, `body`) 파싱 방식으로 전환, API Gateway 응답 형태 제거
- `utils.py` `send_push_notification()`: `firebase_admin` 직접 호출 → `boto3.invoke(InvocationType="Event")`로 워커 호출 형태로 교체
- **가드 추가**: 호출 전 DynamoDB로 `deviceToken` 유무 확인 → 없으면 worker 호출 자체 스킵
- 호출자(`confirm_seat`, `pass_seat`) 코드 변경 없음 - `send_push_notification(user_id, title, body)` 인터페이스 유지

<br>

### 인프라

- `notification_worker` Lambda 신설 (512MB / 15초, `FIREBASE_SERVICE_ACCOUNT_JSON` 환경변수 보유, API 라우트 없음)
- `confirm_seat` / `pass_seat`: 메모리 **128MB 유지**, `FIREBASE_SERVICE_ACCOUNT_JSON` 제거, `NOTIFICATION_WORKER_NAME` 환경변수 주입
- IAM: `lambda:InvokeFunction` 정책 추가 (Resource = notification_worker ARN)

<br>

## 2. 정량 비교

| 항목 | 적용 전 (동기 방식) | 적용 후 (워커 분리) |
| --- | --- | --- |
| **푸시 발송 자체** | **15초 타임아웃 실패** (128MB cold) | **약 4초 정상 완료** (512MB worker cold) |
| 호출자 람다 메모리 | 해결하려면 4배 상향 필요 (128MB → 512MB) | **128MB 유지** |
| 호출자 람다 응답 시간 | 동기 호출이라 푸시 발송 시간만큼 블로킹 | **~50ms 수준** (worker invoke ~5ms + 본 로직) |
| Firebase 자격증명 주입 | 모든 호출자 람다 | worker 한 곳 |
| 푸시 실패 시 본 API 응답 | 영향 있음 (타임아웃 위험) | **무관** (fire-and-forget) |
| 푸시 발송 자동 재시도 | 없음 | **AWS 자동 2회 재시도** |
| 구버전 사용자(deviceToken 없음) 처리 | worker 호출 후 worker가 스킵 | **호출자에서 worker 호출 자체 스킵** |

<br>

## 3. 운영 모니터링 포인트

<br>

### CloudWatch Logs

```
/aws/lambda/jariitnayo-confirm-seat       → "푸시 알림 워커 호출" 로그 존재 확인
/aws/lambda/jariitnayo-notification-worker → "FCM 알림 전송 완료" 로그 존재 확인
```

호출자에 "푸시 알림 워커 호출 실패" 로그가 보이면 IAM 권한 또는 워커 ARN 설정 점검 필요.

<br>

### CloudWatch Metric

| Metric | 의미 | 임계치 제안 |
| --- | --- | --- |
| `AsyncEventsReceived` | worker에 도달한 비동기 이벤트 수 | 호출 발생 시 0보다 큼 |
| `AsyncEventsFailedToDeliver` | worker invoke 자체 실패 건수 | 0이어야 정상 |
| `Errors` (notification_worker) | worker 실행 실패 건수 | 0에 수렴해야 함 |
| `Duration` (notification_worker) | worker cold start + 실행 시간 | 4초 내외 유지 |

<br>

## 4. 잔여 작업 (Follow-up)

- **DLQ 또는 Lambda Destinations 도입 검토**: AWS의 자동 2회 재시도 이후에도 실패한 경우 어디로 보낼지. 운영 데이터가 누적된 후 결정.
- **AWS Lambda Power Tuning**으로 `notification_worker`의 최적 메모리 탐색 (현재 512MB는 임시 결정값).
- **분산 trace 강화**: 호출자 ↔ worker 사이에 `traceId` 동반 전달하여 CloudWatch Insights에서 단일 흐름으로 조회 가능하도록 보강.
- **실측 데이터 누적**: 배포 후 1 ~ 2주 운영 데이터로 본 문서의 정량 비교 표를 실측치로 갱신.
