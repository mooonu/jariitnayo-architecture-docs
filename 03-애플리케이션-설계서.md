# 애플리케이션 아키텍처 설계서

> 최종 수정일: 2026-06-05



## 1. 컴퓨팅 환경 선택 근거

### 결정사항

AWS Lambda를 컴퓨팅 환경으로 선택한다. 

### 결정 근거

**트래픽 패턴과 비용 구조의 일치**

자리있나요 서비스의 트래픽은 출퇴근 시간대에 집중되고 심야에는 거의 없는 불균등한 패턴을 가진다. EC2와 ECS는 유휴 시간에도 인스턴스가 상시 실행되어 비용이 발생한다. Lambda는 요청이 있을 때만 실행되므로 유휴 시간 비용이 0이다.

예상 비용 비교 (초기 운영 기준):

| 서비스 | 월 예상 비용 | 비고 |
|--------|------------|------|
| Lambda | $1 미만 | DAU 100, 사용자당 5~10회 호출 기준 |
| EC2 (t3.small) | $10~15 | 24시간 상시 운영 |
| ECS (동일 스펙) | $20~25 | EC2 동일 스펙 대비 약 $10 추가 |
| RDS (db.t4g.micro) | $25~30 | DB 별도 |

**자동 스케일링**

Lambda는 요청량에 따라 자동으로 동시 실행 수를 조정한다. 출퇴근 피크 시간에 트래픽이 급증해도 별도의 Auto Scaling 설정 없이 처리된다. EC2는 Auto Scaling Group 구성이 필요하고, ECS는 Service Auto Scaling 설정이 필요하다.

**운영 부담 최소화**

EC2는 OS 패치, 프로세스 관리, 배포 자동화, 보안 설정을 직접 관리해야 한다. Lambda는 런타임 이상의 인프라 관리가 AWS에 위임되어 서비스 로직에 집중할 수 있다.

**DynamoDB와의 조합**

Lambda는 상시 실행 서버가 없으므로 DB 커넥션 풀을 유지할 수 없다. RDS를 사용할 경우 Lambda의 요청마다 새 커넥션이 생성되어 커넥션 고갈 문제가 발생할 수 있고, 이를 해결하기 위한 RDS Proxy가 추가로 필요하다. DynamoDB는 HTTP 기반 API로 커넥션 개념이 없어 Lambda와 자연스럽게 결합된다.

**응답 시간 허용치**

서비스 응답 시간 목표를 최대 3초 이내로 설정했을 때, 콜드 스타트가 발생하더라도 허용 범위 내에 있다고 판단

### 트레이드오프

| 항목 | Lambda (현재) | EC2 | ECS |
|------|-------------|-----|-----|
| **유휴 비용** | $0 | $10~15/월 | $20~25/월 |
| **스케일링** | 자동 (즉시) | Auto Scaling Group 설정 필요 | Service Auto Scaling 설정 필요 |
| **콜드 스타트** | 300ms~1s (첫 요청) | 없음 | 없음 (30초~1분 스케일 아웃) |
| **운영 관리** | 런타임 이상 AWS 위임 | OS/배포/보안 직접 관리 | 배포 자동화 쉬움, 오케스트레이션 설정 필요 |
| **DB 커넥션** | DynamoDB와 조합 우수 | 커넥션 풀 유지 가능 | 커넥션 풀 유지 가능 |
| **전환 비용** | ECS/EC2 전환 시 아키텍처 변경 필요 | ECS 전환 용이 | EC2 전환 용이 |

> ECS는 매력적인 선택지였으나, 초기 서버 1대 운영 시 오케스트레이션 기능이 활용되지 않고 동일 스펙 EC2 대비 월 $10 이상 비용 차이가 발생한다. 트래픽이 증가하거나 운영 부담이 체감될 때 EC2 → ECS로 전환하는 것이 적합하다.



## 2. Lambda 아키텍처 설계

### 결정사항

**1 엔드포인트 = 1 Lambda 함수** 원칙을 적용한다. 단일 Lambda 함수에 라우팅 로직을 포함하는 방식을 사용하지 않는다.

현재 19개 함수로 구성된다 (비즈니스 함수 17개 + Authorizer 1개 + 스케줄러 함수 1개):

```
functions/
├── authorizer/          # JWT 검증 — API Gateway Lambda Authorizer
├── auth/                # POST /api/v1/auth/signup  (timeout: 15s)
├── get_seats/           # GET  /api/v1/seats
├── get_all_seats/       # GET  /api/v1/seats/all  ← 마케팅용 전체 좌석 조회 (공개, 인증 없음)
├── get_my_seats/        # GET  /api/v1/seats/me
├── register_seat/       # POST /api/v1/seats
├── update_seat/         # PATCH /api/v1/seats/{seatId}
├── reveal_seat/         # POST /api/v1/seats/{seatId}/reveal
├── confirm_seat/        # POST /api/v1/seats/{seatId}/confirm
├── pass_seat/           # POST /api/v1/seats/{seatId}/pass
├── seat_continue/       # POST /api/v1/seats/sit  ← 하차역 지정 후 즉시 좌석 등록
├── get_train/           # GET  /api/v1/train/identify
├── get_nearby_arrival/  # GET  /api/v1/arrival/nearby  ← 홈화면 GPS 기반 도착 정보
├── get_points_balance/  # GET  /api/v1/points/balance
├── get_points_history/  # GET  /api/v1/points/history
├── get_level/           # GET  /api/v1/level/me
├── purchase_item/       # POST /api/v1/shop/purchase  ← 상점 아이템 포인트 구매
├── delete_user/         # DELETE /api/v1/users/me  (timeout: 15s)
└── reporter/            # EventBridge Rules 스케줄러 — 매일 KST 07:00 가입자 리포트 Discord 전송
```

### 결정 근거

**독립적인 배포 단위**

함수를 분리하면 변경된 함수만 재배포할 수 있다. CI/CD 파이프라인에서 `git diff`로 변경된 함수를 감지하여 해당 함수만 빌드하고 배포한다. 

**독립적인 스케일링**

Lambda는 함수 단위로 동시 실행 수를 제어할 수 있다. 예를 들어 `get_train`(실시간 열차 조회)에 트래픽이 집중될 때 다른 함수에 영향을 주지 않는다. 

**장애 격리**

특정 함수에서 오류가 발생해도 다른 함수는 영향을 받지 않는다. 

**코드 복잡도 관리**

각 `handler.py`는 단일 엔드포인트의 비즈니스 로직만 담당한다.

**환경 변수 관리**

`lambda.tf`를 보면 `get_seats`, `get_train`, `pass_seat`에만 `SEOUL_API_KEY` 환경변수가, `auth`와 `delete_user`에만 `APPLE_*` 환경변수가 추가로 주입된다. 함수가 분리되어 있기 때문에 각 함수에 필요한 환경변수만 선택적으로 주입할 수 있다. 

### 트레이드오프

| 항목 | 1 함수 = 1 엔드포인트 (현재) | Fat Lambda |
|------|--------------------------|-----------|
| **배포 단위** | 변경된 함수만 재배포 | 전체 재배포 |
| **스케일링** | 함수별 독립 스케일링 | 전체 동시 실행 한도 공유 |
| **장애 격리** | 함수 단위 격리 | 라우팅 버그가 전체 영향 |
| **Terraform 코드량** | 함수 수만큼 리소스 정의 증가 | 단일 리소스 정의 |
| **콜드 스타트** | 함수별 독립 웜업 | 단일 함수 웜업 |
| **모니터링** | 함수별 CloudWatch 로그 분리 | 단일 로그 스트림에서 엔드포인트 구분 필요 |

### 2.1 개별 함수 설계 근거

**`get_nearby_arrival`**

홈화면에서 GPS 기반으로 가장 가까운 역의 실시간 도착 정보를 노선별로 표시하는 기능이다. 기존 `get_train`(특정 역의 열차 목록 반환, 검색 화면용)과 응답 구조가 다르다. `get_train`을 재사용하면 핸들러 내부에 분기가 생겨 복잡도가 증가하므로 별도 함수로 분리했다. 문제 발생 시 홈화면 요청인지 열차 검색 요청인지 로그에서 즉시 구분할 수 있다는 운영상 이점도 있다.

- 인증 없이 공개 엔드포인트로 운영한다. 도착 정보는 역 이름만 있으면 누구나 조회 가능한 공공 데이터이며, `get_train`, `get_seats`와 동일하게 `authorization_type = "NONE"`이다.
- `SEOUL_API_KEY` 환경변수를 추가로 주입받는다. `get_train`, `get_seats`, `pass_seat`와 동일한 패턴이다.

**`seat_continue`**

사용자가 지하철에 탑승한 후 "앉았어요" 버튼을 누르면 하차역을 지정하고 현재 자리를 즉시 등록하는 기능이다. 기존 `register_seat`(하차 예정 좌석 등록)와 별도의 흐름으로, 이미 앉아있는 상태에서 좌석 정보를 등록한다. `POST /api/v1/seats/sit` 엔드포인트로 처리한다.

**`get_all_seats`**

마케팅 화면에서 현재 등록된 모든 활성 좌석을 표시하는 기능이다. 인증 없이 공개 엔드포인트로 운영한다. SeatRegistrations 테이블의 `status-index` GSI를 사용하여 ACTIVE 항목만 Query하며, 테이블 전체를 읽는 Scan을 사용하지 않는다. `batch_get_item`으로 등록자 정보를 한 번에 조회하여 N+1 문제를 방지하고 최대 50개로 제한한다. 자세한 의사결정은 `archive/get_all_seats-설계-의사결정.md` 참조.

**`reporter`**

매일 전날 가입자 현황(신규 가입자, 전일 대비 증감, 누적 가입자)을 Discord 웹훅으로 전송하는 내부 리포팅 함수다. API 엔드포인트가 없으며 EventBridge Rules로 매일 KST 07:00에 자동 실행된다.

스케줄링 방식으로 EventBridge Rules(`aws_cloudwatch_event_rule`)와 EventBridge Scheduler(`aws_scheduler_schedule`) 중 Rules를 선택했다. 하루 1회 실행이라는 단순한 요건에는 두 방식 모두 적합하나, Scheduler는 Lambda를 호출할 별도 IAM Role을 추가로 생성해야 한다. Rules는 `aws_lambda_permission`으로 Lambda 리소스에 호출 권한을 부여하는 방식으로 추가 IAM Role 없이 동작하므로 Rules를 선택했다.

**`purchase_item`**

상점 화면에서 포인트로 아이템(기프티콘 등)을 구매하는 기능이다. `ConditionExpression = "points >= :val"`을 활용해 포인트 부족 시 원자적으로 차단하며, `PURCHASE_ITEM` 타입으로 TransactionLogs에 기록한다. `utils.py`의 `record_transaction()`을 사용하지 않고 직접 `put_item`을 호출하는 이유는 `brandName`, `itemName` 등 상점 전용 필드를 추가로 저장해야 하기 때문이다. `SEOUL_API_KEY` 없이 `common_env`만 주입받는다.

**`auth` 및 `delete_user` 타임아웃 15초**

Apple 로그인 지원이 추가되면서 `auth` 함수가 Apple JWKS 엔드포인트(`https://appleid.apple.com/auth/keys`)에서 공개키를 가져오는 과정에서 지연이 발생하여 기본 타임아웃(3초)으로 타임아웃 에러가 발생했다. 이를 해결하기 위해 15초로 상향 조정했다. `delete_user`도 Apple 탈퇴 처리 시 동일한 외부 API 호출이 필요하여 동일하게 적용했다.

```hcl
resource "aws_lambda_function" "auth" {
  timeout = 15  # Apple JWKS 호출 지연으로 인해 3 → 15 상향
}
```



## 3. Lambda Layer 설계

### 결정사항

공통 코드(`utils.py`)와 공통 패키지(`requirements.txt`)를 Lambda Layer로 분리한다. 모든 비즈니스 함수는 이 레이어를 공유한다.

```
common/python/
├── utils.py          # 공유 유틸리티 코드
├── requirements.txt  # 공유 Python 패키지 (httpx, python-jose, boto3 등)
└── common.zip        # 빌드된 레이어 아티팩트
```

```hcl
# lambda.tf
resource "aws_lambda_layer_version" "common" {
  filename            = "../lambda/common/python/common.zip"
  layer_name          = "${local.prefix}-common"
  compatible_runtimes = ["python3.12"]
}
```

`utils.py`가 제공하는 기능:

| 카테고리 | 함수/객체 |
|---------|---------|
| 로거 | `get_logger` (JSON 구조 로깅, CloudWatch Logs Insights 쿼리 가능) |
| 응답 포매터 | `api_response`, `error_response` |
| 요청 파싱 | `parse_body`, `get_query_params`, `get_path_parameter` |
| 인증 | `get_current_user_id`, `TokenError` |
| DynamoDB 테이블 참조 | `users_table`, `seat_registrations_table`, `transaction_logs_table` |
| 비즈니스 유틸 | `build_train_id`, `parse_car_number`, `new_reg_id`, `now_iso` |
| 포인트/레벨 | `record_transaction`, `update_user_points_and_level` |

### 결정 근거

**코드 중복 제거**

16개 함수 모두 DynamoDB 테이블 참조, 응답 포매터, JWT 검증 로직이 필요하다. 레이어 없이 각 함수에 복사하면 동일 코드가 16벌 존재하게 된다. 버그 수정이나 로직 변경 시 모든 함수를 수정해야 한다.

**패키지 중복 배포 방지**

`python-jose`, `httpx`, `boto3` 등의 패키지는 용량이 크다. 레이어로 분리하면 각 함수의 zip 파일에는 `handler.py` 하나만 포함되어 배포 패키지 크기가 최소화된다. Lambda의 배포 패키지 크기 제한(압축 50MB, 비압축 250MB)에 여유가 생긴다.

**비즈니스 로직 일관성 강제**

포인트/레벨 변경은 `update_user_points_and_level()`과 `record_transaction()`을 반드시 함께 호출해야 한다. 이 로직을 레이어의 `utils.py`에 집중시키면 각 함수가 직접 DynamoDB를 조작하는 것을 방지하고 일관된 처리를 강제할 수 있다.

**핸들러 네이밍 통일**

모든 함수의 핸들러를 `handler.handler`로 통일한다. 레이어가 공통 로직을 담당하므로 각 `handler.py`는 엔드포인트 비즈니스 로직에만 집중할 수 있다.

### 함수별 패키지 관리를 선택하지 않은 이유

각 Lambda 함수마다 `requirements.txt`를 따로 두고 패키지를 개별 관리하는 방식은 채택하지 않았다. `python-jose`, `httpx`, `boto3` 같은 공통 패키지가 19개 함수 모두에 필요한데, 함수마다 동일한 패키지를 각각 포함하면 중복이 너무 많아지고 버전 불일치 같은 관리 문제가 생긴다. 레이어 하나로 공통 패키지를 관리하면 패키지 업데이트 시 레이어만 수정하면 된다.



## 4. DynamoDB 선택 근거

### 결정사항

데이터베이스로 DynamoDB를 선택한다. 

### 결정 근거

**Lambda와의 커넥션 호환성**

RDS는 TCP 커넥션 기반이다. Lambda는 요청마다 새 실행 환경이 생성될 수 있어 요청마다 새 DB 커넥션이 생성된다. 동시 요청이 많아지면 RDS의 커넥션 한도를 초과할 수 있고, 이를 해결하기 위해 RDS Proxy가 필요하다(추가 비용 발생). DynamoDB는 HTTP API 기반으로 커넥션 개념이 없어 이 문제가 발생하지 않는다.

**비용 구조**

DynamoDB PAY_PER_REQUEST 과금은 요청이 없으면 비용이 없다. DAU 100 기준 사용자당 5~10회 호출 시 월 $1 미만이 예상된다. RDS db.t4g.micro는 24시간 상시 실행으로 월 $25~30이 발생한다. Lambda의 유휴 비용 절감 이점을 DB에서도 동일하게 가져갈 수 있다.

**캐시 레이어 불필요**

DynamoDB의 단일 항목 조회(`GetItem`) 응답 시간은 10~25ms 수준이다. RDS는 복잡한 쿼리나 트래픽 증가 시 캐시(Redis) 레이어가 필요해질 수 있다. DynamoDB는 별도 캐시 없이도 충분한 응답 속도를 제공한다.
그리고 애초에 캐시 레이어가 필요하지도 않다. (실시간성 보장을 위해)

**접근 패턴 사전 설계 가능**

이 서비스의 조회 패턴은 명확하다: "특정 열차의 좌석 목록", "내가 등록한 좌석", "내 포인트 내역". 패턴이 고정적이므로 DynamoDB의 스키마 설계 제약(PK 기반 조회)이 문제가 되지 않는다.

### 트레이드오프

| 항목 | DynamoDB (현재) | RDS |
|------|---------------|-----|
| **비용** | $1 미만/월 (DAU 100 기준) | $25~30/월 (상시 실행) |
| **Lambda 커넥션** | HTTP API, 커넥션 없음 | TCP 커넥션, RDS Proxy 필요 가능성 |
| **캐시 필요성** | 불필요 (10~25ms) | 트래픽 증가 시 Redis 필요 가능성 |
| **쿼리 유연성** | PK/SK/GSI 기반 제한적 조회 | SQL로 복잡한 조건 처리 가능 |
| **트랜잭션** | 제한적 (TransactWrite 지원) | ACID 트랜잭션 완전 지원 |
| **학습 비용** | NoSQL 설계 패턴 학습 필요 | SQL 경험 그대로 활용 |
| **스케일링** | 자동 (PAY_PER_REQUEST) | 인스턴스 크기 수동 조정 |

> RDS는 팀 모두 SQL 경험이 있어 시작하기 좋은 선택지이나, Lambda 선택 시 비용 절감 이점을 가져가기 어렵고 커넥션 관리 복잡도가 증가한다.



## 5. DynamoDB 스키마 설계

### 결정사항

3개 테이블로 구성한다. 모든 테이블은 `PAY_PER_REQUEST` 과금 방식을 사용한다.

**과금 방식 선택 근거**: 서비스 초기 요청량을 예측할 수 없어 `PROVISIONED`(용량 사전 지정)를 사용하지 않는다. `PAY_PER_REQUEST`는 실제 요청량에 따라 과금되므로 초기 운영 비용을 최소화할 수 있다.

---

### 5.1 Users 테이블

```
PK:  userId (String)
GSI: ssoId-index (PK: ssoId, projection: KEYS_ONLY)
```

| 속성 | 타입 | 설명 |
|------|------|------|
| `userId` | S | PK. 형식: `"kakao#<ssoId>#<uuid4>"` 또는 `"apple#<ssoId>#<uuid4>"` |
| `ssoId` | S | GSI PK. 소셜 로그인 provider의 raw ID (프리픽스 없음) |
| `nickname` | S | 랜덤 생성 익명 닉네임 |
| `points` | N | 현재 포인트 잔액 |
| `level` | N | 현재 레벨 (소수점 포함) |
| `status` | S | 계정 상태: `ACTIVE` / `DELETED` |
| `createdAt` | S | ISO 8601 UTC 생성 시각 |
| `deletedAt` | S | 탈퇴 시각 (탈퇴 시에만 존재) |
| `withdrawalReason` | S | 탈퇴 사유 Enum (탈퇴 시에만 존재) |
| `withdrawalReasonDetail` | S | 기타 탈퇴 사유 직접 입력 (OTHER 선택 시에만 존재) |

**`withdrawalReason` Enum 값**

| 값 | 설명 |
|----|------|
| `INSUFFICIENT_INFO` | 원하는 기능이나 정보가 부족해서 |
| `INCONVENIENT` | 앱 사용이 불편하거나 어려워서 |
| `INACCURATE_REALTIME` | 실시간 정보의 정확도가 아쉬워서 |
| `LOW_USAGE` | 자주 사용하지 않게 되어서 |
| `OTHER` | 기타 (직접 입력 → `withdrawalReasonDetail`에 저장) |

**PK 형식 변경 근거: `"kakao#<kakaoId>"` → `"kakao#<ssoId>#<uuid4>"`**

초기 설계에서 `userId = "kakao#<kakaoId>"`로 구성했을 때, 탈퇴 후 재가입 시 동일한 PK가 생성되어 기존 탈퇴 계정과 충돌하는 문제가 발생한다. DynamoDB의 PK는 변경이 불가능하므로 탈퇴 처리 후 동일 카카오 계정으로 재가입하면 새로운 사용자로 처리할 수 없다.

이를 해결하기 위해 회원가입 시마다 UUID를 부여하여 PK를 고유하게 만든다. 동일 소셜 계정으로 재가입해도 새로운 `userId`가 생성되어 완전히 독립된 계정으로 처리된다.

**`ssoId` GSI 추가 근거 및 `kakaoId` → `ssoId` 통합 결정**

`userId`에 UUID가 포함되면서 JWT 만료 후 재로그인 시 기존 사용자 여부를 `userId`만으로 조회할 수 없게 되었다. 소셜 로그인으로 얻은 ID로 기존 ACTIVE 계정을 찾아야 하므로 GSI가 필요하다.

초기에는 `kakaoId-index`로 설계했으나, Apple 로그인 추가 시 `appleId-index`를 별도로 추가해야 하는 문제가 생긴다. 소셜 provider를 추상화한 `ssoId` 속성으로 통합하면 GSI 하나로 모든 소셜 로그인 조회를 처리할 수 있다.

```
kakao 로그인: ssoId = "1234567"
apple 로그인: ssoId = "000011.9134c48e58484ad52299cb393541aa78.1781"
```

각 provider의 ID는 이미 고유하므로 프리픽스 없이 raw ID를 그대로 저장한다.

**GSI projection을 `KEYS_ONLY`로 설정한 근거**

초기에는 `projection_type = "ALL"`로 설정하여 GSI에 모든 속성을 복제했다. 그러나 GSI는 원본 테이블을 별도 인덱스 테이블로 복제하는 구조이므로, ALL 설정 시 모든 속성이 이중으로 저장되어 스토리지 비용이 증가한다.

실제 조회 패턴을 분석하면 `ssoId-index` GSI의 목적은 `ssoId → userId` 변환이다. `auth` 함수에서 GSI 쿼리 후 `items[0]["userId"]`만 사용하므로 `userId` 이외의 속성은 이 흐름에서 불필요하다. `KEYS_ONLY`로 변경하면 GSI에는 PK(`ssoId`)와 테이블 PK(`userId`)만 저장되어 스토리지를 절약할 수 있다.

```hcl
global_secondary_index {
  name            = "ssoId-index"
  hash_key        = "ssoId"
  projection_type = "KEYS_ONLY"  # ssoId → userId 변환만 필요
}
```

**Soft Delete 채택 근거**

탈퇴 시 데이터를 물리적으로 삭제하지 않고 `status = DELETED`로 표시한다. 비즈니스 분석 목적으로 탈퇴 사유 데이터를 보존하고, 탈퇴 후 재가입 시 이력 추적이 가능하다. Lambda Authorizer에서 `status = DELETED` 계정의 JWT 요청을 401로 차단한다.

**단순 PK (SK 없음) 유지 근거**

Users 테이블은 항상 `userId` 단일 키 또는 `ssoId-index` GSI로 조회된다. 복합 키가 필요한 1:N 관계가 없으므로 SK를 두지 않는다.

### 5.2 SeatRegistrations 테이블

```
PK: trainId (String)
SK: regId (String)
GSI: userId-index (PK: userId, projection: ALL)
GSI: status-index (PK: status, projection: INCLUDE)
TTL: expiresAt
```

| 속성 | 타입 | 설명 |
|------|------|------|
| `trainId` | S | PK. 형식: `lineId + trainNo` (예: `"1004A1065006507"`) |
| `regId` | S | SK. 형식: `"REG_" + 12자리 hex` |
| `userId` | S | GSI PK. 형식: `"kakao#<kakaoId>"` |
| `lineId` | N | 지하철 호선 ID (서울 API 원본값, 예: 1004) |
| `direction` | S | 방향 (예: 상행, 외선순환) |
| `trainNo` | S | 열차 번호 (서울 API 원본값) |
| `currentStation` | S | 현재역 |
| `carNumber` | N | 탑승 칸 번호 (정수) |
| `carNumberStr` | S | 탑승 칸 원본 문자열 (예: "5번 칸") |
| `seatPosition` | S | 좌석 위치 (예: "앞-왼쪽") |
| `destinationStation` | S | 하차 예정역 |
| `status` | S | `ACTIVE` / `CANCELLED` / `CONFIRMED` |
| `expiresAt` | N | TTL Unix timestamp (등록 후 2시간) |
| `createdAt` | S | ISO 8601 UTC 생성 시각 |

**PK 설계: `trainId = lineId + trainNo` 직접 연결**

`trainId` 구성 방식은 세 단계를 거쳐 확정되었다:

```
"4호선#2234"  →  "LINE4_2234"  →  "1004A1065006507"
(한글 문자열)     (가공값)          (서울 API 원본값 그대로)
```

최종적으로 서울시 실시간 지하철 API의 `subwayId`와 `btrainNo`를 가공 없이 그대로 연결한다. 클라이언트가 API 응답값을 그대로 전달하면 서버에서 `build_train_id(lineId, trainNo)`로 PK를 생성할 수 있어 변환 로직이 단순하다. 한글 문자열이나 임의 가공값을 사용하면 API 응답 형식 변경 시 PK가 깨질 위험이 있다.

**복합 키 (PK + SK) 설계 근거**

하나의 열차(`trainId`)에 여러 좌석 등록(`regId`)이 존재할 수 있다. PK만으로는 열차 내 개별 좌석을 식별할 수 없으므로 SK로 `regId`를 추가한다. 이 구조로 "특정 열차의 모든 좌석 목록" 조회를 PK Query 한 번으로 처리할 수 있다.

**GSI (`userId-index`) 설계 근거**

기본 테이블은 `trainId` 기준으로만 빠른 조회가 가능하다. "내가 등록한 좌석"을 조회하려면 `userId` 기준 조회가 필요한데, PK가 아닌 속성으로 조회하면 테이블 전체를 읽는 Scan이 발생한다. Scan은 테이블 크기에 비례하여 비용과 지연이 증가한다.

GSI는 DynamoDB가 내부적으로 유지하는 별도 인덱스 테이블로, `userId`를 PK처럼 사용하는 인덱스를 생성한다. 이를 통해 두 가지 조회를 Scan 없이 처리한다:

- `get_my_seats`: 내가 등록한 좌석 목록 조회
- `register_seat` 중복 체크: 이미 ACTIVE 상태 등록이 있는지 확인

`projection_type = "ALL"`로 설정하여 GSI에서 모든 속성을 조회할 수 있다. 별도 테이블 조회 없이 GSI 결과만으로 응답을 구성할 수 있다.

**`status-index` GSI 추가 근거**

마케팅 화면(`get_all_seats`)에서 현재 ACTIVE 상태인 모든 좌석을 조회해야 한다. 기본 테이블은 `trainId`로만 빠른 조회가 가능하므로 `status` 기준 조회를 위한 별도 GSI가 필요하다. Scan 대신 GSI Query로 처리하여 비용과 지연을 일정하게 유지한다.

`status`처럼 카디널리티가 낮은 속성을 GSI PK로 사용하면 핫 파티션 문제가 발생할 수 있다. 그러나 DAU 100 수준에서 파티션당 초당 3,000 RCU 한도에 도달하려면 동시 접속자 3,000명 이상이 필요하여 현재 단계에서 현실적인 위험이 없다. 자세한 의사결정은 `archive/get_all_seats-설계-의사결정.md` 참조.

projection을 `INCLUDE`로 설정하고 마케팅 화면 응답에 필요한 속성만 포함한다:

```hcl
global_secondary_index {
  name               = "status-index"
  hash_key           = "status"
  projection_type    = "INCLUDE"
  non_key_attributes = ["lineId", "destinationStation", "passedCount", "userId", "expiresAt", "carNumber", "createdAt"]
}
```

`seatPosition` 등 민감한 위치 정보는 포함하지 않는다.

**TTL 설계 근거**

좌석 등록은 실시간 정보이므로 오래된 데이터가 남아있으면 안 된다. `expiresAt`에 등록 시각 + 2시간을 설정하여 DynamoDB가 자동으로 만료된 항목을 삭제한다. 수동으로 만료 항목을 정리하는 배치 작업이 불필요하다.

**`carNumber`와 `carNumberStr` 병행 저장 근거**

클라이언트에서 "5번 칸"과 같은 문자열로 전달되는 칸 번호를 정수(`carNumber = 5`)와 원본 문자열(`carNumberStr = "5번 칸"`) 모두 저장한다. 정수는 정렬이나 범위 비교에 활용하고, 원본 문자열은 UI 표시에 그대로 사용한다.

---

### 5.3 TransactionLogs 테이블

```
PK: userId (String)
SK: timestamp_txId (String)
```

| 속성 | 타입 | 설명 |
|------|------|------|
| `userId` | S | PK. 형식: `"kakao#<ssoId>#<uuid4>"` 또는 `"apple#<ssoId>#<uuid4>"` |
| `timestamp_txId` | S | SK. 형식: `"<ISO8601>_tx_<8자리 hex>"` |
| `type` | S | `SIGNUP` / `SEAT_REGISTER` / `VIEW_SEAT` / `CONFIRM_REWARD` / `PURCHASE_ITEM` |
| `pointChange` | N | 포인트 변동량 (양수: 획득, 음수: 소비) |
| `levelChange` | N | 레벨 변동량 |
| `relatedRegId` | S | 관련 좌석 등록 ID (`PURCHASE_ITEM` 타입에는 없음) |
| `brandName` | S | 브랜드명 (`PURCHASE_ITEM` 타입에만 존재) |
| `itemName` | S | 아이템명 (`PURCHASE_ITEM` 타입에만 존재) |

**복합 키 설계 근거**

`userId`를 PK로 설정하면 특정 사용자의 모든 거래 내역을 단일 Query로 조회할 수 있다. SK를 `timestamp_txId`로 구성하면 시간순 정렬이 자동으로 적용된다. ISO 8601 형식의 타임스탬프는 문자열 정렬이 시간순 정렬과 일치하는 특성을 활용한다.

SK에 타임스탬프만 사용하지 않고 `_tx_<uuid>` 접미사를 추가한 이유는 동일 사용자가 동일 밀리초에 여러 트랜잭션을 생성할 경우 SK 충돌을 방지하기 위해서다.

**추가 전용(Append-Only) 설계**

TransactionLogs는 한 번 기록된 내역을 수정하거나 삭제하지 않는다. 포인트/레벨 변동의 감사 추적(audit trail) 역할을 한다. 현재 잔액은 Users 테이블에서 관리하고, TransactionLogs는 변동 이력만 기록한다.
