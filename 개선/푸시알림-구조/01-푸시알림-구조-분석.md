# 푸시 알림 발송 구조 분석

[2026.06.17]

<br>

## 0. 요약

- 푸시 발송 Lambda가 단순 1회 푸시 발송에 **15초 타임아웃** 초과로 실패. 
  같은 패턴을 쓰는 `confirm_seat`/`pass_seat`에서도 동일 문제 발생할 위험
- 원인은 **무거운 라이브러리(grpc / cryptography) + 작은 메모리(128MB) 조합으로 인한 CPU 스로틀링**
- `notification_worker` 워커 람다로 분리하여 비동기 호출 구조로 전환. 호출자는 128MB 유지, worker만 512MB

<br>

## 1. 문제 상황

`notification_worker` 도입 직전 단계에서 FCM 인프라 검증용으로 만든 `test_push` Lambda를 1회 호출했을 때 **타임아웃 15초 초과로 실행 실패** 발생. 단순히 본인 userId로 본인 디바이스에 푸시 한 건을 발송하는 가벼운 호출이었음에도 정상 응답 없음.

<br>

### 현재 푸시 발송 구조 (문제 구조)

푸시 알림이 API 요청 처리 핵심 비즈니스 로직 Lambda 내부에서 **동기 방식으로 직접 발송**되고 있었음.

- 발송 주체: `confirm_seat`(착석 확인), `pass_seat`(자리 패스), `test_push`(검증용)
- 동작 흐름:
  1. 클라이언트가 API Gateway를 통해 Lambda 호출
  2. Lambda가 DynamoDB 조회/업데이트 수행
  3. 알림 조건 충족 시 `utils.py`의 `send_push_notification()` 호출
  4. 함수 내부에서 `firebase_admin` import + 초기화 → FCM 서버로 TLS 네트워크 요청
  5. **푸시 전송이 완전히 끝나야만 클라이언트에 HTTP 200 응답 반환**

```
[Client]
  │ POST /confirm
  ▼
[confirm_seat Lambda]  메모리 128MB
  ├─ DynamoDB 업데이트
  ├─ send_push_notification()
  │   ├─ firebase_admin import   ← 무거운 모듈 로드
  │   ├─ TLS 핸드셰이크          ← cryptography 연산
  │   └─ FCM 전송 대기            ← 동기 블로킹
  └─ HTTP 200 응답  ← 푸시 발송 끝나야 반환
```

<br>

### 영향 범위

| 함수 | 현재 메모리 | 호출 빈도 | 문제 노출 시나리오 |
| --- | --- | --- | --- |
| `confirm_seat` | 128MB | 자리 양도 확정 시 | cold start에서 응답 지연 / 타임아웃 |
| `pass_seat` | 128MB | 5표 누적 도달 시 | 동일 |
| `test_push` | 128MB | 검증용 | 본 사건의 발견 지점 |

운영 배포 직전 발견하지 못했다면 사용자가 자리 양도 확정 또는 5회 패스 투표 도달 시점에 **API 응답 15초 지연 또는 502 게이트웨이 타임아웃** 경험 가능성.

<br>

## 2. 원인 분석

<br>

### 1단계 - 로그 타임스탬프 간격 분석

`test_push` 핸들러 곳곳에 구간 진입/종료 로그를 심어 각 단계 사이 경과 시간 측정.

| 구간 | 경과 시간 |
| --- | --- |
| 요청 수신 → DynamoDB 조회 완료 | < 100ms |
| **DynamoDB 조회 완료 → Firebase 초기화 시작** | **7.7초** |
| Firebase 초기화 → FCM 전송 요청 | < 100ms |
| **FCM 전송 요청 → 타임아웃 종료** | **6.3초+** |

→ 두 구간(7.7초 + 6.3초)에서 도합 14초 이상 소모되어 15초 제한 안에 응답 불가.

<br>

### 2단계 - 소스코드 대조

7.7초가 걸린 구간의 소스코드는 단 두 줄.

```python
import firebase_admin
from firebase_admin import credentials, messaging
```

"단순 패키지 로드에 8초가 걸린다?" → 비정상 신호. 패키지 의존성 확인:

- `firebase_admin`은 내부적으로 `grpc`, `google-auth`, `cryptography` import
- 각 패키지가 C 확장(`.so`)을 로드하고 OpenSSL 컨텍스트를 초기화
- 모두 **CPU 연산량이 많은 작업**

<br>

### 3단계 - AWS Lambda 아키텍처 한계 (CPU 스로틀링)

Lambda 실행 리포트에서 `Memory Size: 128 MB` 환경 확인.

> AWS Lambda는 메모리 크기에 비례해 CPU 파워가 할당됨.
> 1,769MB일 때 vCPU 1개 풀 할당. 128MB는 그 1/14 수준의 CPU만 제공.

따라서 무거운 import + TLS 핸드셰이크가 **정상 CPU의 ~7% 수준에서 돌아가다 보니** 다음 병목이 동시 발생.

| 구간 | 정상 CPU 환경 | 128MB CPU 스로틀링 | 배수 |
| --- | --- | --- | --- |
| `firebase_admin` import | ~0.5초 | **7.7초** | ~15배 |
| TLS 핸드셰이크 + FCM 전송 | ~0.5초 | **6.3초 이상** | ~12배+ |

→ **결론: 무거운 라이브러리 + 작은 메모리 조합으로 인한 CPU 스로틀링.**

<br>

## 3. 테스트 - 1차 해결 시도 (메모리 상향)

CPU 스로틀링이 원인이므로 `test_push` 람다 메모리를 **128MB → 512MB**로 상향하여 CPU 파워 4배 확보.

| 항목 | 128MB | 512MB | 개선폭 |
| --- | --- | --- | --- |
| `firebase_admin` import 시간 | 7.7초 | **1.7초** | 78% 단축 |
| FCM 통신 및 TLS 암호화 | 6.3초 (타임아웃) | **2.0초** | 정상 완료 |
| 전체 실행 시간 (cold start) | 15초 초과 (실패) | **약 4.0초** | 성공 |

→ 기능은 동작. 그러나 핵심 비즈니스 로직 람다(`confirm_seat`, `pass_seat`)에 그대로 적용하면 **새로운 형태의 비효율** 발생.

<br>

### 1차 해결책의 한계점

**비용 비효율 (오버프로비저닝)**

- `confirm_seat` / `pass_seat`의 본래 책임은 매우 가벼움 (DynamoDB 읽기/쓰기 50ms 이내)
- 알림은 일부 분기에서만 발송 (예: `pass_seat`은 5표 누적된 그 1회만)
- Lambda 요금은 `메모리 × 실행 시간`. 512MB는 128MB보다 초당 단가 **4배** 비쌈
- 4번의 평범한 요청 처리할 때조차, 5번째 호출에서 알림 1번 가능성 때문에 전체를 4배 비싼 메모리로 돌리는 셈

**UX 저하**

메모리를 올려도 cold start 시점에 여전히 1 ~ 2초 지연 잔존. 사용자가 '확인' 버튼 누른 후 푸시 발송 완료까지 화면이 멈춘 채 대기.

**결합도 증가**

Firebase 자격증명(`FIREBASE_SERVICE_ACCOUNT_JSON`)을 모든 호출자 람다에 주입해야 함. 환경변수 누수 표면 증가 + 자격증명 회전 작업 람다 수만큼 반복.

<br>

## 4. 해결

<br>

### 검토한 대안

| 방안 | 효과 | 기각 사유 | 결정 |
| --- | --- | --- | --- |
| 호출자 람다 메모리 일괄 상향 (128MB → 512MB) | 동기 호출 그대로 두고 CPU 확보 | 오버프로비저닝 (4배 비용), cold UX 잔존(1~2초), 자격증명 결합도 증가 | 기각 |
| `firebase_admin` Lambda Layer 분리 | import 시간 단축 시도 | 핵심 비용(CPU 스로틀링)은 그대로. 효과 미미 | 기각 |
| **`notification_worker`로 책임 분리 + 비동기 invoke** | 호출자 128MB 유지, worker 512MB. 응답 50ms 수준 | - | **채택** |

<br>

### 채택 - 책임 분리 (`notification_worker`)

알림 발송을 전담하는 워커 람다를 신설하고, 호출자 람다는 **요청만 던지고 즉시 종료**.

```
[Client]
  │ POST /confirm
  ▼
[confirm_seat]  128MB 유지
  ├─ DynamoDB 업데이트
  ├─ deviceToken 존재 여부 확인
  ├─ boto3.invoke(InvocationType="Event")  ← 큐에 적재 즉시 반환 (~5ms)
  └─ HTTP 200 응답  ← 사용자 응답 50ms 수준

     ↓ AWS 내부 큐
[notification_worker]  512MB
  ├─ firebase_admin import          (콜드 시 1.7초)
  ├─ FCM 전송                       (콜드 시 2.0초)
  └─ 종료
        ↓
   [APNs / Android FCM 채널]
        ↓
   [사용자 디바이스]
```

<br>

### 효과 비교

| 항목 | 1차 (메모리 일괄 상향) | 2차 (책임 분리) |
| --- | --- | --- |
| 사용자 응답 시간 | cold 시 4초 | **항상 50ms 수준** |
| 메모리 비용 | 모든 호출자 람다가 512MB | **호출자 128MB 유지, worker만 512MB** |
| 결합도 | Firebase 자격증명을 호출자에 주입 | 자격증명은 worker 한 곳에만 |
| 실패 격리 | 푸시 실패가 본 API 응답에 영향 | invoke async 실패도 본 API에 무관 |
| 자동 재시도 | 직접 구현 필요 | AWS Lambda가 비동기 호출 실패 시 자동 2회 재시도 |

<br>

### 트레이드오프 (수용 가능 판단)

| 항목 | 영향 | 수용 근거 |
| --- | --- | --- |
| 푸시 도착 1 ~ 4초 지연 | cold 시 worker가 깨어나는 동안 알림이 잠시 늦게 도착 | 수신자는 앱 안에 머물 필요가 없으므로 도착 지연이 UX에 무관 |
| 분산 로그 | 호출자/워커 CloudWatch Log Group 분리 | 추후 traceId 동반 전달로 보강 가능 |
| invoke async 실패 가시성 | fire-and-forget이라 호출자가 결과를 모름 | CloudWatch Metric `AsyncEventsFailedToDeliver` 모니터링 + 추후 DLQ |
