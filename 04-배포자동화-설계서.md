# 배포 자동화 아키텍처 설계서

> 최종 수정일: 2026-06-02



## 1. CI/CD 전체 흐름

### 결정사항

`lambda` 브랜치에 push가 발생하면 GitHub Actions가 자동으로 **변경된 함수만 감지하여 빌드하고 배포**한다.

```
[개발자: lambda 브랜치 push]
        │
        ▼
[GitHub Actions: detect job]
        │  git diff --name-only <before>..<after>
        │  functions/ 하위 변경 파일 필터링
        │  변경된 함수 디렉토리 목록 추출 → JSON 배열
        │
        ▼ (변경된 함수가 없으면 deploy job 스킵)
[GitHub Actions: deploy job (matrix 병렬 실행)]
        │
        ├─ AWS OIDC 인증 (저장된 자격증명 없음)
        │   role-to-assume: arn:aws:iam::{AWS_ACCOUNT_ID}:role/{service}-oidc
        │
        ├─ zip 빌드
        │   cd functions/{function_name}
        │   zip -r {function_name}.zip handler.py
        │
        └─ Lambda 코드 업데이트
            aws lambda update-function-code
              --function-name {service}-{function-name}
              --zip-file fileb://{function_name}.zip
```

**함수명 변환**: 디렉토리명의 언더스코어를 하이픈으로 변환하여 AWS Lambda 함수명에 맞춘다.

```bash
FUNCTION_NAME=$(echo "${{ matrix.function }}" | tr '_' '-')
# get_my_seats → {service}-get-my-seats
```

**matrix 전략**: 변경된 함수 목록을 JSON 배열로 만들어 `strategy.matrix`에 전달한다. 여러 함수가 변경된 경우 병렬로 동시 배포된다.



## 2. Lambda 배포 방식 결정 근거 (zip vs Container 이미지)

### 결정사항

zip 방식으로 Lambda 함수를 배포한다. Container 이미지 방식을 사용하지 않는다.

### 결정 근거

**패키지 크기**

각 함수의 배포 패키지는 `handler.py` 단일 파일이다. 공통 의존성(`python-jose`, `httpx`, `boto3` 등)은 Lambda Layer로 분리되어 있어 함수 zip은 수 KB에 불과하다. zip 방식의 최대 크기 제한(압축 해제 250MB)에 전혀 근접하지 않는다. Container 이미지(최대 10GB)의 대용량 지원이 필요하지 않다.

**배포 속도**

zip 방식은 `aws lambda update-function-code` 한 번으로 배포가 완료된다. Container 이미지 방식은 Dockerfile 빌드 → ECR 푸시 → Lambda 이미지 업데이트의 단계가 필요하고, 이미지 크기에 따라 배포 시간이 길어진다.

**인프라 단순화**

Container 이미지 방식은 ECR(Elastic Container Registry) 리포지토리가 필요하다. zip 방식은 추가 AWS 서비스 없이 Lambda에 직접 업로드한다. 포트폴리오 규모에서 ECR 관리 복잡도를 추가할 이유가 없다.

**Layer와의 조합**

의존성 관리를 Lambda Layer로 처리하고 있어 Container 이미지의 주요 장점인 "Dockerfile에 의존성 포함"이 필요하지 않다.

### 비교

| 항목 | zip 방식 (현재) | Container 이미지 방식 |
|------|--------------|-------------------|
| **최대 크기** | 250MB (압축 해제) | 10GB |
| **배포 속도** | 빠름 | 상대적으로 느림 |
| **의존성 관리** | Lambda Layer 필요 | Dockerfile에 포함 |
| **로컬 테스트** | 상대적으로 불편 | Docker로 동일 환경 재현 가능 |
| **ECR 필요** | 불필요 | 필요 |
| **복잡도** | 단순 | 상대적으로 복잡 |



## 3. Zip 생성 주체 결정 근거

### 결정사항

**GitHub Actions(메인)와 로컬 Terraform(서브) 두 가지 방식을 병행**한다. 
일상적인 코드 변경은 GitHub Actions가 처리하고,
Lambda 최초 생성 시에는 로컬에서 zip을 생성한 후 `terraform apply`를 실행한다.

### 결정 근거

**Lambda 최초 생성 시 제약**

`terraform apply`로 Lambda 함수를 처음 생성할 때 `filename` 경로에 zip 파일이 반드시 존재해야 한다. GitHub Actions는 push 이벤트에 반응하므로 인프라가 아직 없는 최초 생성 시점에는 동작하지 않는다. 따라서 최초 생성은 로컬에서 zip을 먼저 만들고 `terraform apply`를 실행하는 순서가 필수다.

**일상적인 코드 변경은 GitHub Actions**

Lambda 함수 코드를 수정하는 경우 `lambda` 브랜치에 push하면 GitHub Actions가 변경된 함수만 자동으로 빌드하고 배포한다. 개발자가 직접 zip을 만들고 AWS CLI를 실행할 필요가 없다.

**향후 자동화 확장 가능성**

Terraform 상태 파일을 원격 백엔드로 전환하면 GitHub Actions에서 `terraform apply`까지 실행하는 완전 자동화가 가능하다. 현재는 로컬 상태 파일 제약으로 인프라 변경은 수동으로 처리한다.

**역할을 분리한 이유**

Terraform이 `terraform apply`를 실행하면 상태 파일(`terraform.tfstate`)을 읽고 써야 하는데, 현재 로컬에만 있어 GitHub Actions에서 접근할 수 없다. 또한 CI/CD가 `terraform apply`를 실행하면 인프라 드리프트가 있을 경우 코드 변경 외의 의도치 않은 인프라 변경이 발생할 수 있다. 역할을 분리하면 이 두 가지 문제를 모두 피할 수 있다.

**Makefile로 로컬 빌드 단순화**

반복되는 zip 생성 작업을 Makefile로 자동화했다. 함수 추가 시 Makefile에 타겟만 추가하면 된다.

```bash
make all          # 전체 빌드 (layer + 모든 함수)
make layer        # 공통 레이어만
make register_seat  # 특정 함수만
make clean        # zip 전부 삭제
```

### 배포 역할 분담

| 상황 | 담당 주체 | 방법 |
|------|---------|------|
| Lambda 최초 생성 | 로컬 | `make all` → `terraform apply` |
| 코드 변경 배포 | GitHub Actions | `lambda` 브랜치 push |
| 인프라 변경 (라우트 추가 등) | 로컬 | `terraform apply` |
| 레이어 변경 | 로컬 | `make layer` → `terraform apply` |



## 4. Lambda 변경 감지 방식

### 결정사항

CI/CD(GitHub Actions)는 `git diff`로 변경된 함수를 감지하고, Terraform은 zip 파일의 hash 비교로 변경을 감지한다. 두 방식이 각자의 역할에서 병행 사용된다.

### 각 방식의 동작

**GitHub Actions — git diff 방식**

```yaml
CHANGED=$(git diff --name-only ${{ github.event.before }} ${{ github.event.after }} \
  | grep '^functions/' \
  | cut -d'/' -f2 \
  | sort -u \
  | while read func; do
      if [ -d "functions/$func" ]; then echo "$func"; fi
    done \
  | jq -R -s -c 'split("\n") | map(select(length > 0))')
```

이번 push에서 변경된 커밋 범위(`before`~`after`)를 검사하여 `functions/` 하위에서 변경된 디렉토리만 추출한다. 실제로 디렉토리가 존재하는지 확인하는 가드 조건도 포함되어 있다.

**Terraform — zip hash 비교 방식**

```hcl
source_code_hash = filebase64sha256("../lambda/functions/register_seat/register_seat.zip")
```

`terraform apply` 실행 시 현재 zip 파일의 SHA256 hash를 AWS에 배포된 함수의 hash와 비교한다. hash가 다르면 해당 함수만 재배포한다. 코드가 변경되면 zip 내용이 바뀌고 hash도 달라진다.

### 각 방식 비교

| 항목 | git diff (CI/CD) | zip hash (Terraform) |
|------|----------------|---------------------|
| **감지 기준** | 이번 push의 파일 변경 | zip 파일 내용 변경 |
| **배포 방법** | `aws lambda update-function-code` | `terraform apply` |
| **적합한 상황** | 일상적인 코드 변경 | 최초 생성, 인프라 변경 동반 시 |
| **누락 위험** | `before` SHA가 없는 경우 (첫 push) 전체 배포 | zip이 커밋되지 않으면 감지 불가 |

> **`source_code_hash` 제거 검토 중**: Terraform은 최초 생성과 인프라 변경에만 사용하므로 `source_code_hash`가 실질적으로 동작하는 상황이 없다. 오히려 인프라 변경 목적으로 `terraform apply`를 실행할 때 로컬에서 `make`를 실행한 뒤라면 zip이 GitHub Actions가 배포한 버전과 달라져 코드가 의도치 않게 덮어씌워질 위험이 있다. 현재는 로컬 zip이 최초 배포 시점 그대로 유지되고 있어 문제가 발생하지 않고 있으나, 잠재적 위험 요소로 판단하여 제거를 검토 중이다. → `06-13-할일.md` 참조



## 5. OIDC 인증 방식 결정 근거

### 결정사항

GitHub Actions에서 AWS 자격증명을 GitHub Secrets에 저장하지 않는다. OIDC(OpenID Connect)를 통해 임시 자격증명을 발급받아 사용한다.

```yaml
permissions:
  id-token: write   # OIDC 토큰 발급 권한
  contents: read

- name: Configure AWS credentials (OIDC)
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::{AWS_ACCOUNT_ID}:role/{service}-oidc
    aws-region: ap-northeast-2
```

### 결정 근거

**자격증명 유출 위험 제거**

AWS Access Key ID와 Secret Access Key를 GitHub Secrets에 저장하면 Secrets 노출, 레포지토리 설정 오류, 내부자 접근 등의 경로로 유출될 수 있다. OIDC는 장기 자격증명을 저장하지 않고 워크플로우 실행 시 AWS STS에서 임시 자격증명을 발급받는다.

**임시 자격증명의 자동 만료**

OIDC로 발급된 자격증명은 워크플로우 실행 시간 동안만 유효하다. 유출되더라도 짧은 시간 후 자동 만료된다. 저장된 Access Key는 명시적으로 삭제하기 전까지 유효하다.

**신뢰 관계 기반 접근 제어**

IAM Role의 신뢰 정책에서 특정 GitHub 레포지토리와 브랜치만 해당 역할을 assume할 수 있도록 제한할 수 있다. 다른 레포지토리나 브랜치에서는 같은 역할을 사용할 수 없다.

### 비교

| 항목 | OIDC (현재) | 저장된 자격증명 (Access Key) |
|------|-----------|--------------------------|
| **자격증명 저장** | 없음 | GitHub Secrets에 저장 |
| **유효 기간** | 워크플로우 실행 중만 유효 | 명시적 삭제 전까지 유효 |
| **유출 위험** | 낮음 (임시 토큰) | 높음 (장기 자격증명) |
| **설정 복잡도** | IAM Role + 신뢰 정책 설정 필요 | Secrets 등록만으로 간단 |
| **자격증명 교체** | 불필요 | 주기적 교체 권장 |
