# 보안 아키텍처 설계서

> 최종 수정일: 2026-06-12

## 1. 인증 흐름

### 결정사항

Kakao 및 Apple OAuth를 클라이언트(Flutter SDK)에서 처리하고, 서버는 소셜 토큰 검증 후 자체 JWT를 발급한다. 이후 모든 보호 요청은 Lambda Authorizer가 JWT를 검증한다.

```
[Flutter 앱]
    │  1. Kakao SDK 또는 Apple SDK로 OAuth 처리 (클라이언트 측)
    │     → Kakao Access Token 또는 Apple Identity Token 획득
    │
    ▼
[POST /api/v1/auth/signup]  ← Authorizer 없음 (공개 엔드포인트)
    │
    ▼
[Lambda: auth]  (timeout: 15s)
    │  2. 소셜 토큰 검증
    │     Kakao: GET https://kapi.kakao.com/v1/user/access_token_info
    │     Apple: GET https://appleid.apple.com/auth/keys (JWKS 공개키 검증)
    │     → ssoId 획득
    │
    │  3. ssoId-index GSI로 Users 테이블 조회 (KEYS_ONLY)
    │     → userId 획득 후 기존 유저 반환 or 신규 생성
    │     userId = "{provider}#<ssoId>#<uuid4>"
    │
    │  4. 자체 JWT 발급 (HS256, 30일 만료)
    │     payload: { "userId": "kakao#...", "exp": ... }
    │
    ▼
[Flutter 앱: JWT 저장 (shared_preferences)]
    │
    │  5. 이후 모든 보호 요청에 JWT 포함
    │     Authorization: Bearer <JWT>
    │
    ▼
[API Gateway → Lambda Authorizer]
    │  6. JWT 서명 검증 + userId 추출
    │     성공 → userId를 컨텍스트에 주입 → 비즈니스 Lambda 호출
    │     실패 → 403 반환 (비즈니스 Lambda 미호출)
    │
    ▼
[Lambda: 비즈니스 함수]
    │  7. get_current_user_id(event)로 userId 추출
    │     event["requestContext"]["authorizer"]["lambda"]["userId"]
```

**서버에서 소셜 OAuth를 처리하지 않는 이유**

Flutter SDK가 Kakao/Apple OAuth를 클라이언트에서 직접 처리하므로 서버에서 OAuth URL 생성이나 콜백 처리가 불필요하다. 서버는 클라이언트가 전달한 소셜 토큰의 유효성만 검증한다.

## 2. Lambda Authorizer 선택 근거

### 결정사항

API Gateway의 JWT 검증 주체로 Lambda Authorizer를 사용한다. API Gateway 내장 JWT Authorizer를 사용하지 않는다.

### 결정 근거

**JWT Authorizer의 구조적 제약**

API Gateway 내장 JWT Authorizer는 HTTP API 전용 기능으로, JWT 검증 시 외부 JWKS(JSON Web Key Set) URL에서 공개키를 가져와 서명을 확인하는 방식으로 동작한다.

```
JWT Authorizer 동작:
JWT 꺼냄 → JWKS URL에서 공개키 가져옴 → 공개키로 서명 검증
```

이 방식은 반드시 JWKS URL이 필요하다. 자리있나요는 자체적으로 JWT를 발급하는 구조이며, 자체 발급 JWT에는 JWKS URL이 없다. 따라서 JWT Authorizer는 현재 인증 구조와 호환되지 않는다.

**Lambda Authorizer의 유연성**

Lambda Authorizer는 별도 Lambda 함수가 JWT 검증 로직을 직접 구현한다. `python-jose` 라이브러리로 HS256 서명을 검증하고, `JWT_SECRET_KEY` 환경변수의 시크릿 키를 사용한다. 자체 발급 JWT를 자체 시크릿으로 검증하는 구조에 적합하다.

**코드에서 추론한 추가 근거**

`apigateway.tf`에서 `enable_simple_responses = false`로 설정되어 있다. Simple Response는 `isAuthorized: true/false`만 반환하는 간단한 방식이다. 이를 비활성화하고 IAM Policy 형식의 응답을 사용함으로써 `userId`를 컨텍스트에 주입하는 기능을 활용한다. JWT Authorizer는 이런 커스텀 컨텍스트 주입을 지원하지 않는다.

```hcl
resource "aws_apigatewayv2_authorizer" "lambda" {
  authorizer_type                   = "REQUEST"
  authorizer_payload_format_version = "2.0"
  enable_simple_responses           = false  # 커스텀 컨텍스트 주입을 위해 비활성화
}
```

### 비교

| 항목 | Lambda Authorizer (현재) | JWT Authorizer |
|------|------------------------|---------------|
| **자체 발급 JWT 지원** | ✅ | ❌ (JWKS URL 필수) |
| **커스텀 검증 로직** | 가능 | 불가 |
| **컨텍스트 주입** | userId 등 커스텀 데이터 주입 가능 | 제한적 |
| **추가 Lambda 비용** | Authorizer 호출 비용 발생 | 없음 |
| **지연시간** | Authorizer Lambda 실행 시간 추가 <br />(단, 결과 캐싱으로 완화 — 아래 참조) | 없음 |
| **유지보수** | 검증 로직 직접 관리 | AWS 관리 |

### Authorizer 결과 캐싱 (2026-06-12 도입)

#### 배경

CloudWatch `Invocations` 메트릭상 `{service}-authorizer`가 **전체 함수 중 호출량 1위**로, 일 평균 500회 이상(측정일 554회)을 기록했다. 2위 `get-points-balance`(215회)의 약 2.5배다. 보호 라우트(12개) 요청마다 Authorizer Lambda가 매번 실행되고, 그 안에서 Users 테이블 `get_item`(탈퇴 여부 확인)까지 1회씩 수행하기 때문이다. 즉 **인증이 필요한 모든 요청이 Lambda invoke 1회 + DynamoDB 조회 1회를 항상 추가로 부담**하는 구조였다.

#### 결정사항

`aws_apigatewayv2_authorizer`에 `identity_sources`(캐시 키)와 `authorizer_result_ttl_in_seconds`(TTL)를 설정해 Authorizer 결과 캐싱을 활성화한다. 같은 토큰은 TTL 동안 직전 판정(Allow + `userId` 컨텍스트)을 재사용하며, 이 동안 Authorizer Lambda와 Users 조회를 건너뛴다. (거부는 캐싱되지 않음 — authorizer가 예외를 던지므로 Allow만 캐싱된다.) **카나리로 60초부터 시작해 검증 후 300초로 상향**한다.

```hcl
resource "aws_apigatewayv2_authorizer" "lambda" {
  authorizer_type                   = "REQUEST"
  authorizer_payload_format_version = "2.0"
  enable_simple_responses           = false
  identity_sources                  = ["$request.header.Authorization"]  # 캐시 키 = 토큰
  authorizer_result_ttl_in_seconds  = 60                                 # 카나리(검증 후 300으로 상향)
}
```

> 배포 순서 주의: **handler.py(Allow Resource를 wildcard로 변경)를 먼저 배포한 뒤** terraform apply로 캐싱을 켠다. 역순(캐싱 ON + 구버전 핸들러의 라우트별 ARN)이면 캐시된 정책이 다른 라우트에서 Resource 불일치로 403을 일으킨다.

> 참고: 이 캐싱은 02-네트워크-설계서의 "게이트웨이 레벨 캐싱 ❌"(= REST API의 **응답 캐싱**, 실시간 지하철 데이터에 부적합)과는 별개 기능이다. 응답 본문이 아니라 **인증 판정**을 캐싱한다.

#### 추가 변경 — Allow 정책 Resource를 wildcard로

캐싱을 켜면 한 토큰의 판정이 모든 보호 라우트에서 재사용되어야 한다. 기존 `authorizer/handler.py`는 Allow 정책의 Resource로 **특정 라우트 ARN**(`routeArn`)을 반환했는데, 이 경우 `/seats/me`에서 캐시된 정책이 `/points/balance` 요청에는 Resource 불일치로 거부된다. 따라서 Resource를 **해당 스테이지 전체 wildcard**(`arn:...:apiId/$default/*`)로 변경했다.

#### 도입 효과 및 주의사항

| 항목 | 내용 |
|------|------|
| **이득** | 캐시 히트 시 Authorizer Lambda invoke + Users 조회 제거. 콜드스타트 회피. authorizer 호출량이 토큰당 5분 1회로 수렴 |
| **비용/지연** | 보호 API 응답 지연 감소 (`응답지연-분석` 참조) + Lambda 호출 비용 절감 |
| **보안** | 탈퇴(`status=DELETED`) 직후에도 최대 TTL(5분)간 해당 토큰이 통과한다. JWT 자체가 30일 만료이므로 5분 지연은 수용 가능하다고 판단. 즉시 무효화가 필요하면 TTL을 낮추거나 0(캐싱 off)으로 되돌린다 |

## 3. JWT 설계

### 결정사항

HS256 서명 방식, 30일 만료, `userId` 클레임을 사용하는 자체 JWT를 발급한다.

**Payload 구조**

```json
{
  "userId": "kakao#1234567#a1b2c3d4e5f6",
  "exp": 1748000000
}
```

**설계 세부 사항**

| 항목 | 값 | 비고 |
|------|-----|------|
| 서명 알고리즘 | HS256 | 대칭키 서명 |
| 만료 기간 | 30일 | 재로그인 빈도 최소화 |
| 사용자 식별자 클레임 | `userId` | 표준 `sub` 대신 사용 |
| 시크릿 키 관리 | 환경변수 `JWT_SECRET_KEY` | Terraform `var.jwt_secret`으로 주입 |
| 라이브러리 | `python-jose[cryptography]` | — |

**`sub` 대신 `userId` 클레임 사용 이유**

`auth` 함수가 JWT를 발급할 때 `userId` 필드를 사용하고, `authorizer` 함수가 검증할 때도 `payload["userId"]`를 추출한다. 두 함수의 구현 방식을 일치시켜 혼선을 방지한다. 표준 `sub` 클레임을 사용하면 `authorizer`에서 `payload["sub"]`로 추출해야 하는데, 이를 `userId`로 통일하여 코드 일관성을 높였다.

**JWT Secret 환경변수 통일**

초기에는 `JWT_SECRET`과 `JWT_SECRET_KEY`가 혼용되었다. `lambda.tf`를 보면 `authorizer` 함수에는 `JWT_SECRET`, 나머지 함수에는 `JWT_SECRET_KEY`가 주입된다. 이는 `authorizer`가 별도의 환경변수명을 사용하는 레거시 설정이 남아있는 것으로, `JWT_SECRET_KEY`로 통일하는 것이 향후 개선 과제다.

```hcl
# authorizer: 레거시 변수명
environment {
  variables = { JWT_SECRET = var.jwt_secret }
}

# 나머지 함수: 통일된 변수명
environment {
  variables = local.common_env  # JWT_SECRET_KEY 포함
}
```

### 개선해야 할 점

| 항목 | 현재 설계 | 개선 방향 |
|------|---------|---------|
| **서명 방식** | HS256 (대칭키) | RS256 (비대칭키) — 공개키 배포 가능 |
| **만료 기간** | 30일 | Refresh Token 도입 시 Access Token 단기화 |
| **Refresh Token** | 미구현 | 보안 강화 시 도입 |
| **토큰 무효화** | 만료 전 무효화 불가 | 블랙리스트 또는 짧은 만료 기간으로 보완 |

## 4. IAM 기반 접근 제어

### 결정사항

모든 Lambda 함수는 단일 실행 역할(`{service}-lambda-role`)을 사용한다. DynamoDB 접근은 특정 테이블 ARN으로 제한한다. API Gateway의 Lambda 호출 권한은 특정 API ARN으로 제한한다.

**Lambda 실행 역할 구성**

```
{service}-lambda-role
├── AWSLambdaBasicExecutionRole  (CloudWatch Logs 쓰기)
└── {service}-lambda-dynamodb-policy
    └── 허용 액션: GetItem, PutItem, UpdateItem, DeleteItem, Query, Scan
        허용 리소스:
        - {service}-Users ARN
        - {service}-SeatRegistrations ARN
        - {service}-SeatRegistrations GSI ARN (index/*)
        - {service}-TransactionLogs ARN
```

**API Gateway → Lambda 호출 권한 제한**

```hcl
resource "aws_lambda_permission" "register_seat" {
  action        = "lambda:InvokeFunction"
  principal     = "apigateway.amazonaws.com"
  source_arn    = "${aws_apigatewayv2_api.main.execution_arn}/*/*"
  # 이 API Gateway에서 오는 호출만 허용
}
```

`source_arn`을 특정 API Gateway의 ARN으로 제한하여 다른 API Gateway나 직접 호출이 Lambda를 실행할 수 없도록 한다.

**리소스 범위 최소화**

DynamoDB 정책에서 `Resource: "*"` 와일드카드를 사용하지 않고 세 테이블의 ARN을 명시한다. 계정 내 다른 DynamoDB 테이블에는 접근할 수 없다.

현재 구조의 한계: 모든 함수가 동일한 역할을 공유하므로 읽기 전용 함수(`get_level`, `get_points_balance` 등)도 쓰기 권한을 보유한다. 함수별 역할 분리로 최소 권한을 강화할 수 있으나, 19개 함수에 대한 역할 관리 복잡도가 증가한다.

## 5. 미적용 보안 항목

### 5.1 WAF (Web Application Firewall)

**결정사항**: 미적용

**결정 근거**

- 어떤 위협을 막아야 하는지, 어떻게 구성해야 하는지에 대한 이해가 부족한 상태에서 섣불리 도입하기보다 미적용 상태를 명시적으로 인정하는 쪽을 선택했다.
- API Gateway HTTP API는 WAF를 직접 통합할 수 없다. 적용하려면 CloudFront를 앞단에 추가해야 하는 별도 구성이 필요하다.
- DynamoDB를 사용하므로 SQL Injection 자체가 불가능하다.
- CloudFront + WAF 추가 시 월 $5~10의 고정 비용이 발생한다.

**비교**

| 미적용 위험 | 적용 시 이점 |
|-----------|-----------|
| 대규모 요청 공격(DDoS) 방어 없음 | IP 기반 차단, 요청 속도 제한 가능 |
| 악성 페이로드 필터링 없음 | OWASP Top 10 규칙 적용 가능 |
| — | CloudFront CDN 효과 추가 |

### 5.2 KMS (Key Management Service)

**결정사항**: 미적용

**결정 근거**

- DynamoDB는 기본적으로 AWS 관리 키(SSE-S3)로 저장 데이터를 암호화한다. 별도 KMS 설정 없이도 저장 데이터 암호화가 적용된다.
- JWT Secret은 Terraform 변수(`var.jwt_secret`)로 관리되며 Lambda 환경변수로 주입된다. Lambda 환경변수는 AWS 관리 키로 암호화된다.
- KMS 고객 관리 키(CMK) 도입 시 키 교체 정책, 키 접근 감사 등의 추가 관리가 필요하다.
- KMS API 호출 비용이 발생한다 (10,000건당 $0.03).

**비교**

| 미적용 위험 | 적용 시 이점 |
|-----------|-----------|
| 키 교체 자동화 없음 | 고객 관리 키로 암호화 제어권 확보 |
| AWS 관리 키에 의존 | 키 접근 감사 로그(CloudTrail) 상세화 |

### 5.3 Shield (DDoS 방어)

**결정사항**: Shield Standard만 적용 (자동, 무료)

**결정 근거**

- AWS Shield Standard는 모든 AWS 서비스에 자동으로 적용되며 추가 비용이 없다. L3/L4 레벨의 기본 DDoS 방어가 제공된다.
- Shield Advanced는 월 $3,000의 고정 비용이 발생한다. 포트폴리오 규모에서 적용할 수 없다.
- API Gateway는 기본적으로 계정 레벨 요청 한도가 있어 극단적인 트래픽 급증에 대한 기본 보호가 있다.

**비교**

| Shield Standard (현재) | Shield Advanced |
|----------------------|----------------|
| 자동 적용, 무료 | 월 $3,000 |
| L3/L4 기본 방어 | L7 애플리케이션 레벨 방어 |
| 공격 가시성 없음 | 실시간 공격 가시성 + AWS DDoS 대응팀 지원 |
